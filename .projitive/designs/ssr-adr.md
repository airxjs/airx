# SSR 架构决策记录

**任务**: TASK-0015  
**路线图**: ROADMAP-0003  
**版本目标**: 0.8.0+  
**状态**: ACCEPTED

---

## 1. 背景

- **任务**: TASK-0015
- **路线图**: ROADMAP-0003
- **当前问题**: Airx 0.7.0 已支持基础的 Server 端字符串渲染，但缺少状态序列化、客户端激活（hydrate）、上下文传递和错误边界能力
- **目标范围**: 提供完整的 SSR 渲染能力，包括服务端渲染、状态序列化、客户端激活
- **非目标范围**: 框架特定集成（Express/Fastify）、流式渲染、CSS-in-JS 服务端处理、路由级别 SSR

---

## 2. 约束与原则

- **可读性优先**: 代码路径应短、命名直观、模块边界清晰
- **可维护性优先**: 允许后续重构与替换，不锁死在单一实现
- **性能基线**: 不显著退化，暂不追求极限优化
- **兼容性**: 与现有 `createApp`、`Router`、`Signal` 模型协同
- **渐进增强**: 在现有 server.ts 基础上扩展，而非重写

---

## 3. 备选方案

### 方案 A: 扩展现有 server.ts

- **描述**: 在现有的 `render(pluginContext, element, onComplete)` 基础上扩展，新增 `createSSRApp`、`hydrate` 等接口
- **优点**:
  - 复用现有 `ServerElement` 和 `commitDom` 逻辑
  - 最小化代码改动和风险
  - 保持与现有渲染模型的一致性
- **风险**:
  - 现有代码可能有技术债
  - 需要小心不破坏现有功能
- **维护成本**: 低

### 方案 B: 全新 ssr/ 目录

- **描述**: 在 `source/render/ssr/` 下新建独立 SSR 模块，与 `server/` 和 `browser/` 并列
- **优点**:
  - 清晰的模块边界
  - 不影响现有渲染路径
- **风险**:
  - 代码重复（ServerElement 等逻辑需复制）
  - 需要维护多套 commitDom 逻辑
- **维护成本**: 中

### 方案 C: 插件化 SSR

- **描述**: 通过 Plugin 机制支持 SSR，`createApp` 接收 SSR 插件后自动切换到 SSR 模式
- **优点**:
  - 最灵活的扩展方式
- **风险**:
  - 复杂度高，实现难度大
  - 与当前架构差异大，重构成本高
- **维护成本**: 高

---

## 4. 评估矩阵

| 维度 | 权重 | 方案 A | 方案 B | 方案 C |
|---|---:|---:|---:|---:|
| 可读性 | 0.35 | **4** | 3 | 2 |
| 可维护性 | 0.35 | **4** | 3 | 2 |
| 兼容性 | 0.20 | **4** | 3 | 3 |
| 性能基线风险 | 0.10 | **3** | 3 | 3 |
| **加权总分** | | **3.80** | **3.00** | **2.40** |

**评分说明**（1-5 分，5 分最佳）:
- 可读性：方案 A 复用现有代码，路径熟悉；方案 B 需要理解多模块；方案 C 插件机制复杂
- 可维护性：方案 A 改动集中；方案 B 需要同步多份代码；方案 C 重构成本高
- 兼容性：方案 A 自然继承现有设计；方案 B 需要适配层；方案 C 需要大改
- 性能基线：三者相近，方案 A 可能略优（共享代码）

---

## 5. 决策

- **选型结论**: 方案 A — 扩展现有 server.ts
- **关键理由**:
  1. 最低风险：复用现有实现，无需大规模重构
  2. 最高可维护性：代码改动集中，易于追踪和回滚
  3. 符合渐进增强原则：先满足核心需求，再迭代优化
- **放弃原因**:
  - 方案 B：代码重复和维护成本不划算
  - 方案 C：复杂度过高，与当前架构差异大

## 7. 风险与缓解

### 风险 1: Signal ID 机制缺失

- **描述**: 当前 `Signal.State` 无 `id` 字段，无法追踪和恢复状态
- **缓解措施**: 在实现阶段为 `Signal.State` 增加可选 `id` 字段，保持向后兼容

### 风险 2: hydrate 性能问题

- **描述**: 大量组件的 hydrate 可能导致首屏交互延迟
- **缓解措施**: 
  - 首版不追求极限性能
  - 提供 `forceReset` 选项让用户选择
  - 后续可优化（如懒 hydrate）

### 风险 3: 破坏性变更风险

- **描述**: 修改 server.ts 可能影响现有 Server 渲染功能
- **缓解措施**: 
  - 现有 `render()` 函数保持不变
  - 新增 `createSSRApp()` 独立实现
  - 所有修改通过新增文件进行

### 风险 4: 调试体验差

- **描述**: SSR 错误堆栈可能难以追踪
- **缓解措施**:
  - 提供清晰的错误类型和错误码
  - 在错误信息中包含组件栈
  - 文档中说明常见错误处理
