## Code Architecture and Implementation Findings

### 已完成的工作 (0.7.10)

**`source/render/basic/commit-walker.ts`** — commit walker 抽象
- `createCommitWalker(options)` 工厂函数，统一 walk 逻辑
- 复用: `commitWalk(rootInstance, rootDom)` 遍历组件实例树

**`source/render/basic/commit-helpers.ts`** — 共享 DOM 操作辅助函数
- `removeDeletions(instance)` — 移除标删元素 + 触发 unmount 生命周期
- `insertDomIntoParent(instance, domRef, oldNode)` — 插入 DOM 到父节点
- `createPropClassifier()` — 属性分类谓词工厂 (isProperty/isGone/isNew)
- `getDebugElementName(instance)` — 调试友好的元素名称

**共用点 (browser.ts 和 server.ts)**:
- 相同的 `commitInstanceDom` 结构：removeDeletions → 创建 DOM → updateDomProperties → insertDomIntoParent
- 相同的 `createCommitWalker` 调用模式

### 尚未收敛的差异点

**`updateDomProperties` 实现差异**:

| 方面 | Browser | Server |
|------|---------|--------|
| 文本更新 | `plugin.updateDom` | 手动设置 `nodeValue` |
| 属性更新 | `plugin.updateDom` | 手动 `setAttribute`/`removeAttribute` |
| 样式更新 | `plugin.updateDom` | 手动遍历 style 对象 |
| 事件监听 | `plugin.updateDom` | 不支持（SSR 无 DOM 事件） |
| 插件钩子 | ✅ 调用 `plugin.updateDom` | ❌ 不调用 |

**关键发现**: Browser 的 `updateDomProperties` 完全委托给插件系统 (`plugin.updateDom`)，而 Server 是手动实现。这是因为 Server 不需要事件监听，且输出是字符串而非真实 DOM。

**DOM 创建差异**:
- Browser: 使用 `document.createElementNS/createElement/createTextNode/createComment`
- Server: 使用 `ServerElement.createElement/createTextNode/createComment`

这些是平台相关 API，无法进一步抽象。

## Risks and Open Questions

**Q1**: 是否需要将 `createPropClassifier` 的分类规则文档化到 `code-style.md`？
- 当前: 规则内联在 `createPropClassifier()` 中
- 建议: 补充 JSDoc 即可，不需要独立文档

**Q2**: Server 的 `updateDomProperties` 是否应该调用插件钩子？
- 当前: Server 不调用任何插件的 `updateDom`
- 分析: SSR 输出是字符串，插件的 DOM 操作（事件绑定等）在 SSR 场景无意义
- 结论: 当前行为正确，不需要改动

**Q3**: Iteration B 的 "收敛 DOM 创建逻辑" 是否完成？
- 分析: DOM 创建依赖平台 API（document vs ServerElement），无法跨平台共享
- 结论: 架构上已收敛到 `commitInstanceDom` 统一框架，平台差异是不可避免的
