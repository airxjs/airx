## Implementation Findings

### Bug Fixed: performUnitOfWork Re-render Logic Bypass

**Problem**: The early return `if (instance.child) { return instance.child }` at common.ts:596-597 was preventing re-render logic from executing when `instance.child` was already set from the first render.

**Root Cause**: On first render of a reactive component:
1. Component returns a render function
2. `reconcileChildren` is called, creating `instance.child`
3. `instance.child` is set

On subsequent re-render with `needReRender=true`:
1. The code checked `instance.child` first
2. Since `instance.child` was truthy, returned early
3. Re-render logic never executed

**Fix Applied** (common.ts):
```typescript
// Before:
if (instance.child) {
  return instance.child
}

// After:
if (instance.child && !instance.needReRender) {
  return instance.child
}
```

### Additional Fix: needReRender Cleanup

Changed `delete instance.needReRender` to `instance.needReRender = false` for cleaner boolean semantics (undefined vs false).

### Test Corrections

1. **Test 1** (`should create signalWatcher for reactive function components on first render`): 
   - `performUnitOfWork` returns `instance.child` (the div from reconciliation), NOT null
   - Updated expectation to `expect(nextInstance).toBe(instance.child)`

2. **Test 3** (`should re-render with new Signal values when needReRender is set`):
   - For reactive components (components that return functions), the **component function** is called ONCE
   - On re-render, the **render function** (returned by component) is called, not the component itself
   - `componentCallCount` stays at 1, but `componentReturnValue()` is called again
   - Updated test to verify this behavior correctly

### Key Insight: Reactive Component Pattern

In airx's reactive component pattern:
- Component function (e.g., `CounterComponent`) is called ONCE on first render
- Component returns a render function: `() => createElement(...)`
- `instance.componentReturnValue` = that render function
- On re-render, `componentReturnValue()` is called (not the component)
- The render function reads Signals, creating reactive dependencies

## Code Architecture and Implementation Findings

### Signal Watcher Integration（common.ts:533-558）

**关键代码段**：
```typescript
// common.ts:531-558
if (instance.signalWatcher == null) {
  const signalWatcher = signal.createWatch(() => {
    instance.needReRender = true
    onUpdateRequire?.(instance)
    const pendings = signalWatcher.getPending()
    for (const pending of pendings) pending.get()
    signalWatcher.watch()
  })
  instance.signalWatcher = signalWatcher
  instance.context.addDisposer(() => signalWatcher.unwatch())
}
// ...
instance.signalWatcher.watch(childrenComputed)
```

**要点**：
- Watcher 是惰性创建的，只有当 Signal 被读取时才触发
- `childrenComputed` 是 `createComputed(() => componentReturnValue())`
- notify 回调中调用 `getPending()` 消费所有待处理 Signal，再重新 watch
- dispose 时调用 `unwatch()` 清理

### provide/inject 变化检测（inject.ts:17-29）

```typescript
isReuseInstance(instance: Instance): false | undefined {
  const injectedKeys = [...instance.context.injectedMap.keys()]
  for (const key of injectedKeys) {
    const currentValue = instance.context.injectedMap.get(key)
    const parentValue = this.getProvideValueForParent(instance.parent, key)
    if (parentValue !== currentValue) return false
  }
}
```

### Instance 结构（common.ts:231-248）

```typescript
export interface Instance<E extends AbstractElement = AbstractElement> {
  domRef?: E
  child?: Instance<E>
  parent?: Instance<E>
  sibling?: Instance<E>
  deletions?: Set<Instance<E>>
  element?: AirxElement
  beforeElement?: AirxElement
  signalWatcher?: signal.Watcher
  componentReturnValue?: AirxComponentRender | AirxElement
  needReRender?: boolean
  elementNamespace?: string
  context: InnerAirxComponentContext<E>
  memoProps?: any
}
```

### InnerAirxComponentContext provide/inject（common.ts:67-115）

- `provide(key, value)` 存入 `providedMap`
- `inject(key)` 从父级 chain 查找 `providedMap`，返回最近父级的值
- 注入值变化时触发实例不复用（由 `InjectSystem.isReuseInstance` 判断）

### reconcileChildren 实例复用逻辑（common.ts:310-344）

- `isReuseInstance(instance, nextElement)` 由插件决定是否复用
- 复用时更新 `instance.element` 和 `instance.memoProps`
- 不复用时创建新 `Instance`，触发 `context.onMounted`

### 生命周期触发链

- `triggerMounted()` → 调用 `mountListeners`，一次性清理
- `triggerUnmounted()` → 递归调用子/兄弟 `triggerUnmounted`，最后调用 `dispose()`
