## 核心发现

### 1. commitWalkV2 — 完全重复

**位置**: browser.ts (lines ~150-180) vs server.ts (lines ~330-370)

**代码对比**:
```typescript
// Browser 版本
function commitWalkV2(initInstance: Instance<BrowserElement>, initNode?: ChildNode) {
  const stack: StackLayer[] = [[initInstance, initNode]]
  while (stack.length > 0) {
    const stackLayer = stack.pop()!
    if (typeof stackLayer === 'function') { stackLayer(); continue }
    const [instance, node] = stackLayer
    commitInstanceDom(instance, node)
    stack.push(() => instance.context.triggerMounted())
    if (instance.sibling != null) { /* push sibling */ }
    if (instance.child != null) { /* push child */ }
  }
}

// Server 版本 - 结构完全一致
function commitWalkV2(initInstance: Instance<ServerElement>, initNode?: ServerElement) {
  const stack: StackLayer[] = [[initInstance, initNode]]
  while (stack.length > 0) { /* 完全相同的逻辑 */ }
}
```

**重叠度**: 100% — 算法、遍历顺序、生命周期触发时机完全一致

**可抽象接口**:
```typescript
interface CommitNode {
  domRef: HTMLElement | ServerElement | null
  element?: AirxElement
  sibling?: CommitNode
  child?: CommitNode
  deletions?: Set<CommitNode>
  beforeElement?: AirxElement
  context: InnerAirxComponentContext
}

type CommitWalkFn = (root: CommitNode, rootDom?: HTMLElement | ServerElement) => void
```

### 2. commitInstanceDom — 结构性重复，实现差异在 updateDomProperties

**相同逻辑**:
1. **删除处理** — 两者完全一致:
   ```typescript
   if (nextInstance.deletions) {
     for (const deletion of nextInstance.deletions) {
       const domList = getChildDoms(deletion)
       for (const dom of domList) {
         if (dom?.parentNode) dom.parentNode.removeChild(dom)
       }
       deletion.context.triggerUnmounted()
     }
     nextInstance.deletions.clear()
   }
   ```

2. **DOM 创建** — 结构一致，仅构造函数不同:
   ```typescript
   if (nextInstance.domRef == null) {
     if (nextInstance.element.type === INTERNAL_TEXT_NODE_TYPE) {
       // browser: document.createTextNode()
       // server: ServerElement.createTextNode()
     } else if (nextInstance.element.type === INTERNAL_COMMENT_NODE_TYPE) {
       // browser: document.createComment()
       // server: ServerElement.createComment()
     } else {
       // browser: document.createElement()
       // server: ServerElement.createElement()
     }
   }
   ```

3. **Parent 插入逻辑** — 完全一致（含 insertBefore 优化）:
   ```typescript
   if (nextInstance.domRef != null) {
     if (oldNode !== nextInstance.domRef) {
       const parentDom = getParentDom(nextInstance)
       // 错误检查（server 多一个 text/comment 检查）
       if (nextInstance.domRef.parentNode === parentDom) {
         parentDom.insertBefore(nextInstance.domRef, oldNode ?? null)
       } else {
         if (nextInstance.domRef.parentNode) {
           nextInstance.domRef.parentNode.removeChild(nextInstance.domRef)
         }
         parentDom.appendChild(nextInstance.domRef)
       }
     }
   }
   ```

**差异点 — updateDomProperties**:
- **Browser**: 委托给 `plugin.updateDom(dom, nextProps, prevProps)` — 插件系统负责
- **Server**: 内联实现，手动处理 style/class/attributes/textContent

---

### 3. 共享基础设施

以下函数来自 `common.ts`，两者共享:
- `getParentDom()` — 获取 DOM 父节点
- `getChildDoms()` — 获取子 DOM 列表
- `performUnitOfWork()` — 协调工作单元（逻辑相同，泛型参数不同）
- `InnerAirxComponentContext` — 组件上下文类

---

### 4. Browser 独有逻辑

- `isWorkLoopScheduled` 调度标志
- `requestIdleCallback` 空闲时间调度
- `onUpdateRequire` / `scheduleWorkLoop` 调度机制
- `needCommitDom` 提交标记

这些与 Browser 的增量更新机制相关，Server 不需要。

## 收敛方案建议

### 方案 A：提取共享的 CommitWalker（推荐）

提取通用的 commit walk 算法，通过依赖注入解决平台差异:

```typescript
// source/render/basic/commit-walker.ts
interface DOMOperations<T> {
  createElement(tag: string): T
  createTextNode(text: string): T
  createComment(text: string): T
  removeChild(parent: T, child: T): void
  appendChild(parent: T, child: T): void
  insertBefore(parent: T, newNode: T, refNode: T | null): void
  updateProperties(dom: T, nextProps: Props, prevProps: Props): void
}

function createCommitWalker<D>(ops: DOMOperations<D>) {
  return function commitWalk(
    rootInstance: Instance<D>,
    rootDom: D | undefined,
    pluginContext?: PluginContext  // browser 需要
  ) {
    // 统一的 commitWalkV2 逻辑
  }
}
```

**优点**:
- 消除 90% 的重复代码
- 保持各自平台特有的 updateDomProperties 实现
- 调度逻辑保持独立

**风险**:
- 需要修改公共 API 接口
- 需要充分测试确保 SSR 输出不变

### 方案 B：仅文档化差异，保持现状

适合作为短期方案，长期仍有技术债累积风险。
