# TASK-0047 Implementation Research Brief

## Task
审计 airx README.md 与代码导出的一致性

## Summary
对比 source/index.ts 实际导出与 README 描述的 API，确保 README 中的示例与代码一致（如 createApp vs createSSRApp、renderToHTML vs renderToString 等）。

## 代码审查结果

### 1. ✅ `renderToString` API 签名错误 — 已修复
- **问题**: README API Reference 将 `renderToString` 描述为独立函数 `airx.renderToString(app)`
- **实际**: `renderToString()` 是 `SSRApp` 的实例方法 `app.renderToString()`
- **修复**: 更新 README 中的 API Reference 说明和示例

### 2. ✅ Hydration 状态说明过时 — 已修复
- **问题**: README 多处标注 Hydration "planned for 0.8.x" 或 "stub"
- **实际**: Hydration 已在 TASK-0033 中实现（0.7.x）
- **修复**: 移除所有 "planned for 0.8.x" 标注，更新为已实现状态

### 3. ⚠️ Signal 导入说明不完整 — 已部分修复
- **问题**: README Quick Start 使用 `new Signal.State(0)` 但未说明 Signal 来源
- **实际**: airx 运行时通过 `getSignal()` 从 `globalNS['Signal']` 读取 Signal（浏览器环境为 `self.Signal`）
- **当前状态**: 已添加 `import { Signal } from 'signal-polyfill'`，但该模块需要用户手动加载并挂载到全局
- **说明**: 这是框架设计问题，airx 不强制依赖 Signal 的安装方式，用户需自行确保 Signal 全局可用

### 4. ✅ hydrate API 签名 — 一致
- `hydrate(ssrHtml, container, app)` 与 `source/render/server/server.ts` 导出的函数签名一致

### 5. ✅ SSRApp 实例方法 — 一致
- `createSSRApp(element)` 返回带有 `renderToString()` 实例方法的 `SSRApp`
- README Quick Start SSR 示例正确使用了 `app.renderToString()`
