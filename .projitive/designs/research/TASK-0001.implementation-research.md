## Code Architecture and Implementation Findings

### Source Files
- `source/index.ts` - Public API surface (~110 lines):
  - Re-exports all public types and functions
  - `AirxApp` interface: mount(), plugin(), renderToHTML()
  - `createApp()` factory function

- `source/app/app.ts` - App creation (~95 lines):
  - `source/app/app.ts#L1-L30` - AirxApp interface definition
  - `source/app/app.ts#L32-L95` - `createApp()` implementation
    - Creates `PluginContext` for plugin system
    - `ensureAsElement()` converts components to elements
    - `mount()` → `browserRender()` with PluginContext
    - `renderToHTML()` → `serverRender()` with PluginContext
    - `plugin()` → registers plugins on PluginContext

- `source/element/element.ts` - Virtual DOM and component types (~500 lines):
  - Component type definitions: `AirxElement`, `AirxComponent`, `AirxChildren`
  - `createElement()` - Creates virtual DOM elements
  - `component()` - Higher-order function for creating components with lifecycle hooks
  - `Fragment` - Grouping component without DOM nodes
  - `AirxComponentContext` - Component instance context with props, mounted state
  - `createErrorRender()` - Error boundary rendering
  - `isValidElement()` - Element validation

- `source/render/basic/hooks/hooks.ts` - Component context hooks (~200 lines):
  - `provide(key, value)` - Provides value down component tree
  - `inject(key)` - Retrieves provided value from tree
  - `onMounted(listener)` - Registers mount callback
  - `onUnmounted(listener)` - Registers unmount callback
  - `currentComponentContext` - Thread-local component context

- `source/render/basic/common.ts` - Core rendering logic (~300 lines):
  - `reconcile()` - Virtual DOM diffing algorithm
  - `processComponent()` - Component instantiation and rendering
  - `createWatcher()` - Signal watcher for reactivity
  - `scheduleUpdate()` - Batched update scheduling

- `source/render/browser/browser.ts` - Browser renderer (~200 lines):
  - `browserRender()` - DOM rendering implementation
  - `insertHTML()` - Safe HTML insertion
  - Browser-specific DOM operations

- `source/render/server/server.ts` - SSR renderer (~200 lines):
  - `serverRender()` - String-based rendering
  - `SSRApp` interface: `renderToString()`
  - `createSSRApp()` - Creates SSR app
  - `hydrate()` - Hydration stub (0.8.x planned)

- `source/render/basic/plugins/index.ts` - Plugin system:
  - `Plugin` interface - `install(ctx: PluginContext): void`
  - `PluginContext` - Plugin registration and management

- `source/signal/signal.ts` - Signal integration (~50 lines):
  - Thin wrapper around global `Signal` with error handling
  - `getSignal()` - Lazy Signal loader with version check
  - Re-exports Signal.State, Signal.Computed, Signal.Effect

- `source/jsx-runtime.ts` - JSX automatic runtime exports
- `source/jsx-dev-runtime.ts` - JSX dev runtime exports
- `source/symbol/symbol.ts` - Internal symbols (provider key, etc.)
- `source/logger/logger.ts` - Internal logging

### Key Architecture Decisions

1. **Signal-driven reactivity**: Uses TC39 Signal proposal primitives
2. **Virtual DOM diffing**: `reconcile()` compares old/new virtual trees
3. **Component model**: Components are functions returning render functions (not hooks)
4. **Plugin system**: `PluginContext` holds registered plugins, called during render
5. **provide/inject context**: Thread-local `currentComponentContext` for synchronous provide/inject
6. **SSR support**: Separate server renderer that outputs HTML strings
7. **Hydration planned**: `hydrate()` stub exists but not yet implemented (0.8.x)
8. **Zero dependencies**: Only `csstype` as runtime dependency

### External Dependencies
- `csstype@^3.1.3` - CSS property types for TypeScript

### Internal Dependencies
- TC39 Signal proposal (native or `signal-polyfill`)

## Risks and Open Questions

1. **Hydration not implemented**: SSR apps can't hydrate on client yet (0.8.x)
2. **Thread-local context**: `currentComponentContext` uses a global variable - potential issue in concurrent SSR
3. **Plugin system is minimal**: No clear plugin API documentation yet
