# TASK-0052 Implementation Research Brief

**Task**: 改进 hydrate 中 SSR 节点的事件绑定能力
**Summary**: Derived from TASK-0051 completion report known limitation: SSR 节点在 hydrate 后不会自动绑定事件。需要在 connectInstanceTreeToDom 后对需要绑定事件的节点调用 updateDom，使 SSR 输出的交互元素能正常工作。

---

## Design Guidelines and Specs

- `source/render/browser/browser.ts` — Normal browser render flow with `updateDomProperties` in `commitDom`. Key method at `browser.ts#L71-79`: iterates plugins and calls `plugin.updateDom(dom, nextProps, prevProps)`
- `source/render/browser/hydrate.ts` — Current hydrate implementation. `connectInstanceTreeToDom` at lines ~100-130 only sets `domRef` and `airxInstance`, never calls `updateDom`
- `source/render/basic/plugins/internal/basic/basic.ts` — `BasicPlugin.updateDom` at line 61, binds events by iterating props starting with "on" (e.g., `onClick`)
- `source/render/browser/hydrate.integration.test.ts` — Test file with event binding tests at lines ~245-270. Currently skipped because events don't fire.

---

## Code Architecture and Implementation Findings

### Current Hydrate Flow (source/render/browser/hydrate.ts)

1. `hydrate()` builds instance tree via `performUnitOfWork`
2. `connectInstanceTreeToDom()` walks tree, sets `domRef` and `airxInstance` on each DOM node
3. `setupReactiveUpdates()` starts reactive system for updates

**Missing step**: After `connectInstanceTreeToDom`, there is no call to `plugin.updateDom()` to bind event handlers from element props.

### Normal Browser Render Flow (source/render/browser/browser.ts)

In `commitDom()`:
```typescript
function updateDomProperties(dom: BrowserElement, nextProps: PropsType, prevProps: PropsType = {}) {
  for (const plugin of pluginContext.plugins) {
    if (plugin.updateDom) plugin.updateDom(dom, nextProps, prevProps)
  }
}
```

Then in `commitInstanceDom`:
```typescript
// 更新属性
if (nextInstance.domRef != null && nextInstance.element != null) {
  updateDomProperties(
    nextInstance.domRef,
    nextInstance.element.props,
    nextInstance.beforeElement?.props
  )
}
```

### Root Cause

`hydrate.ts` uses `connectInstanceTreeToDom` which only:
1. Sets `instance.domRef` = matching DOM node
2. Sets `(dom as any).airxInstance = instance`

It never calls `plugin.updateDom(dom, props, {})` which is what actually binds events.

## Implementation Plan

### Step 1: Modify `connectInstanceTreeToDom` to accept pluginContext

Change signature from:
```typescript
function connectInstanceTreeToDom(
  appInstance: Instance<HTMLElement>,
  container: HTMLElement
): void
```
To:
```typescript
function connectInstanceTreeToDom(
  pluginContext: PluginContext,
  appInstance: Instance<HTMLElement>,
  container: HTMLElement
): void
```

### Step 2: After setting domRef, call updateDom for each instance

In `walkAndConnect`, after setting `instance.domRef`:
```typescript
if (childDom) {
  instance.domRef = childDom as HTMLElement
  // eslint-disable-next-line @typescript-eslint/no-explicit-any
  ;(childDom as any).airxInstance = instance
  
  // NEW: Bind events via plugins
  if (instance.element) {
    type PropsType = Record<string, unknown>
    for (const plugin of pluginContext.plugins) {
      if (plugin.updateDom) {
        plugin.updateDom(childDom, instance.element.props as PropsType, {})
      }
    }
  }
}
```

### Step 3: Update hydrate() to pass pluginContext

Change:
```typescript
connectInstanceTreeToDom(appInstance, container)
```
To:
```typescript
connectInstanceTreeToDom(pluginContext, appInstance, container)
```

### Step 4: Un-skip the event binding test

In `hydrate.integration.test.ts`, update the event binding test to expect events to fire:
```typescript
const button = container.querySelector('button')
if (button) {
  button.click()
  expect(clickHandler).toHaveBeenCalled()  // Now should work!
}
```

## Risks and Open Questions

- **Risk**: Calling `updateDom` with `{}` as prevProps means "remove all old events" — but since SSR nodes were created without event binding (they were server-rendered), this should be safe (no old events to remove).
- **Assumption**: SSR nodes don't have any existing event listeners that would conflict. Since they were created by `renderToString`, they have no JavaScript event handlers attached.
- **Alternative considered**: Creating a separate `bindEventsToInstanceTree` function called after `connectInstanceTreeToDom`. Decided against this because it would require two tree walks; integrating into `connectInstanceTreeToDom` is more efficient.
