## Code Architecture and Implementation Findings

### 现有渲染管线

#### 1. render/basic（协调引擎）

**位置**: `source/render/basic/common.ts`

**职责**:
- `Instance` 树结构维护（parent/child/sibling）
- `performUnitOfWork` 单步执行与遍历推进
- 组件上下文与生命周期管理
- Signal 响应式追踪

**关键代码**:
```typescript
// performUnitOfWork - 组件树遍历
function performUnitOfWork<T extends AbstractElement>(
  pluginContext: PluginContext,
  instance: Instance<T>
): Instance<T> | null
```

#### 2. render/browser（浏览器渲染）

**位置**: `source/render/browser/browser.ts`

**职责**:
- `workLoop` 消费 `nextUnitOfWork`
- `commitDom` 把 Instance 树变化提交到真实 DOM
- 事件监听器绑定
- `requestIdleCallback` 调度优化

**关键代码**:
```typescript
// browser.ts#L180
export function browserRender(pluginContext: PluginContext, app: AirxApp)
```

#### 3. render/server（服务端渲染）

**位置**: `source/render/server/server.ts`

**职责**:
- `ServerElement` 模拟 DOM API
- `commitDom` 将 Instance 树提交到 ServerElement 树
- `toString()` 输出 HTML 字符串

**关键代码**:
```typescript
// server.ts#L130
export function render(pluginContext: PluginContext, element: AirxElement, onComplete: ServerCompleteHandler)
```

### 渲染管线差异分析

| 方面 | Browser | Server |
|------|---------|--------|
| DOM API | 真实 DOM | ServerElement 模拟 |
| 提交策略 | 增量更新 | 全量构建 |
| 事件绑定 | 是 | 否 |
| 生命周期 | onMounted/onUnmounted | 不触发 |
| 调度器 | workLoop + requestIdleCallback | 同步全量 |

### 同构需求分析

#### 可同构的部分
1. **组件函数执行**: 组件函数本身是纯函数，可以在 Server 和 Browser 执行
2. **Element 创建**: `createElement` 是纯函数
3. **performUnitOfWork**: 协调逻辑复用
4. **Signal 响应式**: Signal 在两端行为一致

#### 不可同构的部分
1. **DOM 提交**: Browser 使用真实 DOM，Server 使用 HTML 字符串
2. **副作用**: 事件绑定、动画、CSS-in-JS
3. **生命周期**: onMounted 在 Server 下不触发
4. **浏览器 API**: `window`、`document`、`localStorage` 等

### Signal 集成约束

**当前 Signal 实现**（`source/signal/signal.ts`）:
- `Signal.State`: 可读写，需要支持序列化
- `Signal.Computed`: 依赖追踪，SSR 时可能多次计算
- `Signal.Effect`: 副作用，SSR 下不执行

**序列化需求**:
```typescript
// Signal.State 需要支持
interface SignalState {
  id: string  // 需要新增
  value: unknown
}
```

## SSR 场景清单

### 场景 1: 纯内容渲染（无交互）

```tsx
// 最简单的 SSR 场景
const Article = component((props: { title: string; content: string }) => {
  return () => createElement('article', null, [
    createElement('h1', null, props.title),
    createElement('div', null, props.content)
  ])
})
```

**特点**:
- 无 Signal 状态
- 无用户交互
- 完全静态内容
- hydrate 可选（可以直接重新渲染）

### 场景 2: 带状态的动态内容

```tsx
const UserProfile = component(() => {
  const user = inject<User>('user')
  return () => createElement('div', null, user?.name ?? 'Guest')
})
```

**特点**:
- 有 Signal 状态
- 需要 hydrate 恢复状态
- 状态序列化必需

### 场景 3: 带交互的单页应用

```tsx
const Counter = component(() => {
  const count = new Signal.State(0)
  return () => createElement('button', {
    onClick: () => count.set(count.get() + 1)
  }, count.get().toString())
})
```

**特点**:
- 有用户交互
- hydrate 必须
- Signal 状态恢复必需
- 事件绑定必需

### 场景 4: 带错误边界的页面

```tsx
const ErrorBoundary = component((props: { error: Error }) => {
  return () => createElement('div', { class: 'error' }, props.error.message)
})

const RiskyComponent = component(() => {
  throw new Error('Something went wrong')
})
```

**特点**:
- 渲染可能出错
- 需要错误边界捕获
- 错误状态需要序列化

## 首版 SSR 支持范围

### 支持

| 场景 | 优先级 | 说明 |
|------|--------|------|
| 纯内容渲染 | P0 | 最基础能力 |
| Signal 状态序列化 | P0 | hydrate 必需 |
| hydrate 客户端激活 | P0 | 交互恢复 |
| SSRContext 上下文传递 | P1 | 请求级别数据 |
| 顶层错误边界 | P1 | 错误兜底 |

### 不支持（首版）

| 场景 | 原因 |
|------|------|
| 框架集成 | 需要框架特定适配器 |
| 流式渲染 | 复杂度高，首版不追求 |
| 路由级别 SSR | 需要路由支持 |
| CSS-in-JS SSR | 需要样式处理方案 |

## 兼容性边界

### 与 createApp 的兼容边界

| API | SSR 支持 | 说明 |
|------|---------|------|
| createApp | ✅ 不适用 | SSR 使用 createSSRApp |
| createSSRApp | ✅ 新增 | SSR 专用 |
| .mount() | ❌ 不支持 | SSR 无 DOM 挂载 |
| .use(plugin) | ❌ 不支持 | SSR 无插件扩展点 |
| .renderToString() | ✅ 新增 | SSR 专用 |

### 与 Signal 的兼容边界

| API | SSR 支持 | 说明 |
|------|---------|------|
| Signal.State | ✅ 支持 | 需支持序列化 |
| Signal.Computed | ✅ 支持 | SSR 可计算 |
| Signal.Effect | ❌ 不支持 | SSR 无副作用 |

### 与生命周期的兼容边界

| 生命周期 | SSR 支持 | 说明 |
|---------|---------|------|
| onMounted | ❌ 不触发 | Server 无挂载 |
| onUnmounted | ❌ 不触发 | Server 无卸载 |
| provide/inject | ✅ 支持 | 上下文传递 |

## Risks and Open Questions

### 已识别风险
1. **Signal ID 机制缺失**: 状态序列化需要追踪 Signal 实例
2. **hydrate 性能**: 大量组件的 hydrate 可能影响首屏交互
3. **同构复杂性**: 某些组件可能在 Server 和 Client 行为不一致

### 开放问题
1. **调试体验**: SSR 错误堆栈如何提供足够的上下文？
2. **测试策略**: jsdom 环境下 hydrate 测试能覆盖多少场景？
3. **文档优先级**: API 文档 vs 使用指南哪个优先？
