## Code Architecture and Implementation Findings

### Public API Chain (source/index.ts)

**source/index.ts** (public entry) re-exports from:
- `./types/index.js` → `./types/types.js` - JSX/DOM types only
- `./render/index.js` → `./render/basic/plugins/index.js` - **only `type Plugin` re-exported**
- `./app/index.js` → `./app/app.js` - `AirxApp`, `createApp`
- `./element/index.js` → `./element/element.js` - `Fragment`, `component`, `createElement`, types
- `./render/index.js` - `inject`, `provide`, `onMounted`, `onUnmounted`
- `./render/index.js` - `SSRApp`, `createSSRApp`, `renderToString`, `hydrate`

### Key Finding 1: `AirxApp.plugin()` and `AirxApp.renderToHTML()` marked `@deprecated WIP`

**File**: `source/app/app.ts#L10-L15`

```typescript
export interface AirxApp {
  mount: (container: HTMLElement) => AirxApp

  /** @deprecated WIP */
  plugin: (...plugins: Plugin[]) => AirxApp
  /** @deprecated WIP */
  renderToHTML: () => Promise<string>
}
```

These two methods are documented as WIP and deprecated in JSDoc but are still part of the public type interface. The `plugin` and `renderToHTML` methods DO exist on the returned object (they work, they're just not fully stable).

### Key Finding 2: `hydrate()` is a stub (no-op)

**File**: `source/render/server/server.ts#L446-L451`

```typescript
hydrate(_container: HTMLElement): void {
  // 初步版本暂不实现，实际使用时 hydrate 是客户端行为
}
```

The `hydrate` function is exported as a public SSR API but is actually a no-op stub. The README (Line 138-144) says hydration is "planned for 0.8.x".

### Key Finding 3: Element module exports more than public API

**File**: `source/element/index.ts` exports:
`Props`, `AirxComponentRender`, `ReactiveComponent`, `isValidElement`, `AirxComponentMountedListener`, `AirxComponentUnmountedListener`, `createErrorRender`

These are NOT re-exported in `source/index.ts` (public API), so not directly exposed to users. However, contributors might use these by importing directly from `airx/element`.

### Key Finding 4: Internal render chain exports

**File**: `source/render/basic/common.ts` exports:
- `INTERNAL_TEXT_NODE_TYPE`, `INTERNAL_COMMENT_NODE_TYPE`
- `Disposer`, `InnerAirxComponentContext`
- `AbstractElement`, `Instance`
- `reconcileChildren`, `performUnitOfWork`

These are exported through the render chain but NOT re-exported in public `source/index.ts`. Architecture doc explicitly says these are internal.

### Key Finding 5: `serverRender` and `browserRender` are aliased internal APIs

**File**: `source/render/index.ts`
```typescript
export { render as serverRender } from './server/index.js'
export { render as browserRender } from './browser/index.js'
```

These are internal functions exposed under different names. Not re-exported in public API.

## Implementation Plan

### Audit Report Structure

1. **Public API Surface Summary** - what users can actually import from `airx`
2. **Issue 1**: `@deprecated WIP` methods in `AirxApp` interface
   - `plugin()` and `renderToHTML()` need clearer communication
   - Options: mark with `@hidden` JSDoc, or remove from type (but keep on object)
3. **Issue 2**: `hydrate()` stub misleads users
   - Need to either: (a) document clearly it's a 0.8.x feature, or (b) throw if called
4. **Issue 3**: Internal module exports
   - Not a user-facing issue but worth documenting for contributors
5. **Recommendations for 0.8.x**

---

## Risks and Open Questions

1. **Should `plugin()` and `renderToHTML()` be removed from the public type?**
   - They work but are marked WIP
   - Removing from type would break existing usage
   - Keeping with `@deprecated` might discourage adoption

2. **How should `hydrate()` be handled?**
   - Currently a silent no-op
   - Could throw `UnimplementedError` to avoid confusion
   - Or add runtime warning when called

3. **Should internal exports be restricted?**
   - Could use TypeScript's `internal` module attribute
   - Or use `/** @internal */` JSDoc comments
   - Would affect contributors more than end users
