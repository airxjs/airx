## Code Architecture and Implementation Findings

### 当前 SSR 实现（source/render/server/server.ts）

#### 入口函数
```typescript
// server.ts#L130
export function render(pluginContext: PluginContext, element: AirxElement, onComplete: ServerCompleteHandler)
```
- 接收 PluginContext、AirxElement、回调函数
- 无公开导出，需通过 app 层调用

#### ServerElement 类（L17-L103）
- 模拟 DOM API：`createElement`、`createTextNode`、`createComment`、`appendChild`、`removeChild`、`setAttribute`、`toString`
- `toString()` 输出 HTML 字符串
- 支持 style、class、attributes 序列化

#### 渲染流程
1. 创建 `rootInstance` 和 `appInstance`
2. `performUnitOfWork` 遍历组件树（与 Browser 共用协调引擎）
3. `commitDom` 将 Instance 树提交到 ServerElement 树
4. `toString()` 输出 HTML

#### commitDom 实现（L138-L290）
- `updateDomProperties`: 更新 DOM 属性（style、class、其他 attributes）
- `getParentDom`: 查找实例的父 DOM
- `getChildDoms`: 查找实例下所有一级 DOM
- `commitInstanceDom`: 提交单个 Instance 的 DOM 变化
- `commitWalkV2`: 深度优先遍历 Instance 树并提交

#### 关键问题
- **无状态序列化**: Signal 状态未序列化，无法在客户端恢复
- **无 hydrate 机制**: 仅支持纯 SSR 字符串渲染，无客户端激活
- **无上下文传递**: 无法在 SSR 时注入请求级别上下文
- **无错误边界**: 组件渲染错误直接抛出，无兜底策略

### 模块职责分析

| 模块 | 职责 | 位置 |
|------|------|------|
| `ServerElement` | 模拟 DOM API，构建 HTML 字符串 | server.ts#L17-L103 |
| `render` | SSR 入口，协调渲染流程 | server.ts#L130 |
| `commitDom` | Instance → ServerElement 提交 | server.ts#L138-L290 |
| `performUnitOfWork` | 组件树遍历协调 | basic/common.ts |

### 与 Browser 渲染的重复
- `commitDom` 逻辑在 browser.ts 和 server.ts 均有实现，约 60% 相似度
- 属性更新逻辑（style、class、attributes）完全重复
- 生命周期触发逻辑部分重复

## Implementation Plan

### 阶段 1: 总体架构设计（本文档）

#### SSR 渲染流程时序
```
[Server] renderToString(app)
  -> createSSRPluginContext(requestContext)
  -> performUnitOfWork(serverPluginContext, appElement)
  -> commitDom(serverElements)
  -> serializeServerElements() 
  -> return { html: string, stateSnapshot: SerializedState }

[Client] hydrate(dom, app, stateSnapshot)
  -> deserializeState(stateSnapshot)
  -> restoreSignalState(snapshot)
  -> createBrowserPluginContext()
  -> performUnitOfWork(browserPluginContext, appElement)
  -> commitDom(browserDoms)
```

#### 模块职责划分
```
source/
├── render/
│   ├── server/
│   │   ├── server.ts         # 纯 SSR 渲染（已有）
│   │   ├── serialize.ts      # 新增：状态序列化
│   │   └── context.ts        # 新增：SSR 上下文传递
│   ├── browser/
│   │   ├── browser.ts        # 纯浏览器渲染（已有）
│   │   └── hydrate.ts        # 新增：客户端激活
│   ├── basic/
│   │   ├── common.ts         # 协调引擎（已有）
│   │   └── commit.ts         # 新增：通用 commitDom 提取
```

#### 核心 API 设计
```typescript
// SSR 入口
export function renderToString(
  element: AirxElement,
  context?: SSRContext
): Promise<SSRResult>

// 客户端激活
export function hydrate(
  dom: Element,
  element: AirxElement,
  stateSnapshot?: SerializedState
): AirxApp

// SSR 上下文
interface SSRContext {
  request?: Request
  headers?: Record<string, string>
  // 扩展点：用户可注入自定义上下文
}

// 状态快照
interface SerializedState {
  signals: Record<string, SignalSnapshot>
  version: string
}
```

### 阶段 2: 关键设计决策（后续任务）

#### 决策点 1: 状态序列化策略
- **方案 A**: 在组件渲染时自动收集被读取的 Signal，生成快照（侵入性低但需运行时支持）
- **方案 B**: 用户显式标记需要序列化的 Signal（侵入性高但可控）
- **推荐**: 方案 A + 显式 API 补充

#### 决策点 2: hydrate 时机
- **方案 A**: 同步 hydrate（首屏快但可能阻塞主线程）
- **方案 B**: 异步 hydrate（不阻塞但可能有闪烁）
- **推荐**: 方案 A，首版不追求极限优化

#### 决策点 3: 错误边界
- **方案 A**: 每个组件独立错误边界（粒度细但复杂度高）
- **方案 B**: 顶层统一错误边界（简单但恢复粒度粗）
- **推荐**: 方案 B，后续可扩展为组件级

## Risks and Open Questions

### 已识别风险
1. **Signal 状态序列化**: 当前 Signal 实现无内置序列化机制，需新增
2. **hydrate 性能**: 大量组件的 hydrate 可能导致首屏交互延迟
3. **路由集成**: SSR 下的路由与 Signal 互操作未验证（architecture-review-0.7.md#L515）

### 开放问题
1. **框架集成**: 是否支持 Express / Fastify / Hono 等主流框架的中间件？
2. **流式渲染**: 是否需要支持流式 SSR（Streaming）？
3. **样式注入**: CSS-in-JS 在 SSR 下的处理策略？
