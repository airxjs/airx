## 2. Code Architecture and Implementation Findings

### 2.1 当前 Airx Signal 封装

**文件**: `source/signal/index.ts`

Airx 封装了 `signal-polyfill`，提供四个 API：

```typescript
createWatch(notify)           // → new Signal.subtle.Watcher(notify)
createState<T>(initial)      // → new Signal.State(initial)
createComputed<T>(fn)        // → new Signal.Computed(fn)
isState<T>(target)           // → target instanceof Signal.State
```

**关键实现细节**:
- `source/signal/index.ts#L20-L34`: `getSignal()` 延迟加载全局 Signal 实例
- `source/signal/index.ts#L36-L41`: `createWatch` 返回 `Watcher` 实例，notify 回调由调用方提供
- 异常处理：Signal 未定义 / 多实例时会抛出明确错误

### 2.2 渲染层与 Signal 响应

**文件**: `source/render/browser.ts`

渲染通过 `browserRender(appContext, element, container)` 实现：
- 挂载时清空容器 `container.innerHTML = ''`
- 依赖 Signal 的响应式更新驱动 UI 变化

**潜在问题**: 当前渲染未接入 Watcher，无法自动响应 Signal 变化（这可能是设计决策或未完成工作）。

### 2.3 signal-polyfill vs TC39 提案 语义差异

| 维度 | signal-polyfill | TC39 Proposal | Airx 当前状态 |
|------|-----------------|---------------|--------------|
| `Signal.State` | ✅ 实现 | ✅ 提案核心 | ✅ `createState` |
| `Signal.Computed` | ✅ 实现 | ✅ 提案核心 | ✅ `createComputed` |
| `Signal.subtle.Watcher` | ✅ 实现 | ✅ 提案核心 | ✅ `createWatch` |
| `Watcher#getPending()` | ✅ | ✅ 提案核心 | ❌ 未暴露 |
| `effect()` | ✅ (polyfill 扩展，非提案) | ❌ 不在提案中 | ❌ 未实现 |
| 变化通知时机 | 同步（在 notify 回调中） | 同步 | 依赖 polyfill |
| 批量更新 | 依赖外部 scheduler | 由实现定义 | 未处理 |
| Decorator `@signal` | ✅ | ✅ | ❌ 未实现 |

**关键发现**:
1. **变化收集**: `Watcher#getPending()` 是关键 API，Airx 当前 `createWatch` 未暴露此能力
2. **响应处理**: `effect()` helper 是 polyfill 扩展，不在 TC39 提案中，Airx 未实现
3. **批量更新**: polyfill 提供了基于 `queueMicrotask` 的批量示例，但需要框架自行实现 scheduler

## 3. Implementation Plan

### 3.1 语义对齐决策

**需要明确的问题**（规划阶段）:

1. **Airx 是否需要实现自己的 `effect()` 语义？**
   - 选项 A：自己实现 effect（参考 polyfill 示例），在浏览器渲染中集成
   - 选项 B：保持轻量，不实现 effect，由用户自行使用 Watcher API
   - **建议**：选项 A，作为框架响应式体验的核心能力

2. **Watcher#getPending() 是否需要暴露？**
   - 选项 A：暴露，支持高级用户自定义调度
   - 选项 B：封装在内部，保持 API 简洁
   - **建议**：选项 B，但内部保留对 `#pending` 的访问能力

3. **批量更新策略**
   - 当前 polyfill 示例使用 `queueMicrotask`
   - 是否需要自定义 scheduler？
   - **建议**：先使用 polyfill 默认行为，后续按需优化

### 3.2 单测覆盖边界定义

**核心模块必须覆盖**:

| 模块 | 覆盖内容 | 优先级 |
|------|---------|--------|
| Signal 封装 | `createWatch`, `createState`, `createComputed`, `isState` 的正常/异常路径 | P0 |
| Watcher 行为 | dirty signal 检测、notify 调用、getPending() 返回 | P0 |
| Computed 求值 | 依赖追踪、自动求值、副作用 | P0 |
| State 读写 | get/set/update、初始值、多次 set | P0 |
| 渲染集成 | 响应式更新、批量更新、错误处理 | P1 |
| 生命周期钩子 | provide/inject/onMounted/onUnmounted | P1 |
| 错误边界 | createErrorRender、异常状态展示 | P2 |

### 3.3 测试框架选择

- **推荐**: Vitest（与 Vite 集成好，TypeScript 支持好，性能优秀）
- **备选**: Jest（如需要更广泛的社区生态）

**建议先在 `package.json` 中增加测试脚手架**:
```json
{
  "scripts": {
    "test": "vitest",
    "test:run": "vitest run",
    "coverage": "vitest run --coverage"
  }
}
```
