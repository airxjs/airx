3. `scripts/compare-benchmark.js` - Regression detection script
   - Lines 35-57: `checkRegression()` function - determines if metric regressed
   - Key constraint: Currently only handles timing metrics (latency/opsPerSec)

4. `benchmark/bench-runner.mjs` - Runtime performance benchmarks
   - Lines 1-14: Imports and setup

   - Key constraint: `JsonBaselineStore.toJSON()` produces flat key-value object

5. `benchmark/bench-framework.mjs` - Benchmark infrastructure
   - Lines 1-15: `BenchmarkResult` class with timing fields
   - Lines 130-135: `JsonBaselineStore` class - simple Map-to-JSON converter

### Key Architecture Decisions

- **Bundle size metric**: Unlike timing metrics (lower is better for latency), bundle size has no inherent "ops/sec". For comparison purposes, we'll use `1/size * 1000` as a synthetic opsPerSec (higher = better, meaning smaller bundle is better). Alternatively, store as `size` field and handle specially in comparison.
- **Decision**: Store bundle size as `{ name, size: bytes, sizeKB: string, ...baselineFields }` so compare-benchmark.js can detect it and apply size-specific regression logic (current > baseline × threshold = regression).

## Implementation Plan

### Step 1: Add bundle size collection to bench-runner.mjs

Add after build completes, before running Signal benchmarks:

```javascript
// Bundle size benchmark
import { readFileSync } from 'fs'
import { join } from 'path'

const outputDir = join(process.cwd(), 'output')
const indexMin = join(outputDir, 'index.min.js')
const stats = fs.statSync(indexMin)
const bundleSize = stats.size

// Save as benchmark metric (reuse BenchmarkResult structure with size field)
const bundleResult = {
  name: 'Bundle Size',
  size: stats.size,
  sizeKB: (stats.size / 1024).toFixed(1) + 'KB',
  // Fake timing fields for JsonBaselineStore compatibility
  mean: stats.size, stddev: 0, median: stats.size, min: stats.size, max: stats.size,
  iterations: 1, opsPerSec: 1000 / stats.size  // opsPerSec = 1/size (higher = smaller = better)
}
```

### Step 2: Update compare-benchmark.js for size metrics

Add detection for `size` field and size-specific regression check:

```javascript
const isSize = metricName === 'Bundle Size' || metricName.includes('size')
if (isSize) {
  // For size: current > previous × (1 + threshold) = regression
  change = calcChange(current.size, previous.size)
  isRegressed = change > threshold * 100
}
```

### Step 3: Run benchmark to establish baseline

```bash
npm run benchmark  # builds + runs benchmarks + saves to benchmark-results.json
```

### Step 4: Commit baseline on main

When merged to main, `benchmark.yml` CI already commits updated baseline.

## Risks and Open Questions

1. **Threshold selection**: 5% bundle size growth is reasonable for a major version, but might be too strict if a new feature is intentionally added. Consider making threshold configurable via `BUNDLE_SIZE_THRESHOLD` env var (default 0.05 = 5%).
2. **Multiple bundles**: Currently only tracks `index.min.js`. Should we also track `index.umd.min.cjs` if it gets built? Currently skipped because it doesn't exist in output.
3. **Minification variation**: esbuild output can vary slightly between runs due to non-determinism. Use median of 3 runs if concerned, or accept ±100 bytes variation.
4. **Tree-shaking changes**: If 0.10.0 intentionally adds APIs, bundle growth is expected and should be reviewed manually before merge.
