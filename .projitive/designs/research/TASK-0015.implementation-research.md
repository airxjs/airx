## Code Architecture and Implementation Findings

### 当前 Server 渲染实现

```typescript
// source/render/server/server.ts
export function render(
  pluginContext: PluginContext, 
  element: AirxElement, 
  onComplete: ServerCompleteHandler
)
```

**问题**:
1. 无状态序列化能力
2. 无 hydrate 机制
3. 无上下文传递
4. 无错误边界

### 候选方案评估

| 方案 | 描述 | 复杂度 | 维护成本 |
|------|------|--------|----------|
| 方案 A: 扩展现有 server.ts | 在现有 `render` 基础上扩展 | 低 | 低 |
| 方案 B: 全新 ssr/ 目录 | 新建独立 SSR 模块 | 中 | 中 |
| 方案 C: 插件化 SSR | 通过 Plugin 机制支持 SSR | 高 | 高 |

**方案 A** 是最实际的选择：
- 复用现有 `ServerElement` 和 `commitDom` 逻辑
- 最小化代码改动
- 符合渐进增强原则

### 首版落地范围

**支持**:
- `createSSRApp` / `renderToString` 渲染接口
- Signal 状态序列化
- `hydrate` 客户端激活
- SSRContext 上下文传递
- 顶层错误边界

**不支持**（非目标）:
- 框架特定集成（Express / Fastify）
- 流式渲染
- CSS-in-JS 服务端处理
- 路由级别 SSR

## Risks and Open Questions

### 已识别风险
1. **Signal ID 机制缺失**: 需要在 Signal.State 中增加 id 字段
2. **hydrate 性能**: 大量组件的 hydrate 可能导致首屏交互延迟
3. **调试体验**: SSR 错误堆栈可能难以追踪

### 开放问题
1. **发布策略**: SSR 功能是否放入 0.8.0 主版本？
2. **包体积**: SSR 相关代码会增加 bundle 大小多少？
3. **文档优先级**: 用户文档 vs API 文档哪个先？
