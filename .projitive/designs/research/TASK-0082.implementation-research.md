| Metric | Description | Target Threshold |
|--------|-------------|-----------------|

- `createWatch(notify)` - Signal监听器创建, line 17

### Key Performance-Sensitive Code Paths

1. **Render Loop** (browser.ts:103-123):
   - Uses `requestIdleCallback` for yielding
   - Checks `deadline.timeRemaining() < 1` for early yield

2. **Signal Reactivity** (common.ts:438-470):
   - Uses `queueMicrotask` for batching updates
   - `signalWatcher.watch()` for reactive tracking

3. **SSR State Management** (`source/render/server/server.ts`):
   - `clearSSRSignals()` - 清理SSR状态, line 447
   - State serialization via `injectStateSnapshotIntoHTML()`

## 3. Implementation Plan

### Phase 1: Benchmark Framework Setup
1. Create `benchmark/` directory in project root
2. Create `benchmark/bench-framework.ts` with:
   - `BenchmarkRunner` class for running benchmarks
   - Statistical analysis (mean, stddev, median)
   - Warm-up and iteration count configuration

### Phase 2: Render Throughput Benchmark
1. Create `benchmark/render-throughput.ts`
2. Measure: Component tree mounting speed
3. Metric: Components rendered per second
4. Configuration:
   - Vary component tree depth (10, 100, 1000 nodes)
   - Vary component complexity (simple div vs complex component)
   - Report: mean time, ops/sec, stddev

### Phase 3: Signal Update Latency Benchmark
1. Create `benchmark/signal-latency.ts`
2. Measure: Time from Signal.State.set() to DOM update
3. Metric: End-to-end latency in milliseconds
4. Configuration:
   - Single component with signal-driven rendering
   - Vary update frequency (1, 10, 100 updates/sec)
   - Report: latency p50, p95, p99

### Phase 4: SSR Speed Benchmark
1. Create `benchmark/ssr-speed.ts`
2. Measure: renderToString() execution time
3. Metric: Average time per render
4. Configuration:
   - Typical app structure (nested components, text content)
   - Vary output size (1KB, 10KB, 100KB HTML)
   - Report: mean time, renders/sec

### Phase 5: CI Integration
1. Add `benchmark` script to package.json
2. Create `.github/workflows/benchmark.yml`:
   - Run benchmarks on PR/push to main
   - Compare against baseline metrics
   - Fail if regression > 10%

## 5. Risks and Open Questions

### Known Risks
1. **Environment variability**: jsdom performance differs from real browsers
   - Mitigation: Use relative comparisons, not absolute values
2. **Warm-up variability**: JIT compilation affects early iterations
   - Mitigation: Include 10-iteration warm-up before measurement
3. **Memory pressure**: Long-running benchmarks may accumulate garbage
   - Mitigation: Run garbage collection between iterations

### Open Questions
1. Should benchmarks include bundle size measurement? (Currently TASK-0081 covers this)
2. What baseline metrics should be recorded for 0.9.0?
   - Decision: Record first-run metrics as baseline, document in benchmark README
