## Code Architecture and Implementation Findings

### 1. `getChildDoms` 函数 (lines 95-121) - 完全未覆盖

```typescript
function getChildDoms(instance: Instance<BrowserElement>): BrowserElement[] {
  const domList: BrowserElement[] = []
  const todoList: Instance<BrowserElement>[] = [instance]

  while (todoList.length > 0) {
    const current = todoList.pop()

    // 找到真实 dom 直接提交
    if (current?.domRef != null) {
      domList.push(current.domRef)
    }

    // 有子节点但是无真实 dom，向下继续查找
    if (current?.domRef == null) {
      if (current?.child != null) {
        todoList.push(current.child)
      }
    }

    // 可能有兄弟节点，则需要继续查找
    if (current?.sibling != null) {
      todoList.push(current.sibling)
    }
  }

  return domList
}
```

**问题**: 函数定义被访问 0 次（基于 f[7]=0）。测试通过 `commitInstanceDom` 间接调用，但 `getChildDoms` 内部的 while 循环和分支从未被执行。

**原因分析**: `getChildDoms` 仅在 `commitInstanceDom` 的 `deletions` 处理分支中被调用（line 152-173）。当前测试场景不涉及标记删除的组件实例，因此该路径从未触发。

### 2. `onUpdateRequire` 函数 (lines 269-275) - 未覆盖

```typescript
function onUpdateRequire() {
  if (context.nextUnitOfWork == null && context.rootInstance.child) {
    context.nextUnitOfWork = context.rootInstance.child

  scheduleWorkLoop()
}
```

**问题**: `onUpdateRequire` 在 `performUnitOfWork` 执行组件更新时被调用。当前测试仅覆盖初次渲染（`createApp`），未触发更新场景。

**需要**: Signal/状态更新触发的更新渲染测试。

### 3. `scheduleIdleWork` 函数 (lines 29-36) - 未覆盖

```typescript
function scheduleIdleWork(callback: (deadline: IdleDeadline) => void) {
  if (typeof globalThis.requestIdleCallback === 'function') {
    globalThis.requestIdleCallback(callback)
    return
  }
  globalThis.setTimeout(() => callback(createIdleDeadline()), 0)
}
```

**问题**: `scheduleIdleWork` 在 `scheduleWorkLoop` 中被调用，但 `requestIdleCallback` mock 了立即执行。测试运行时 `scheduleIdleWork` 的 else 分支（setTimeout 降级）未被覆盖。

### 4. `commitInstanceDom` 删除路径 (lines 152-173) - 部分未覆盖

删除逻辑在以下场景触发：
- 组件返回不同元素（key 变化）
- 组件返回 `false`/`null`（删除旧节点）

当前测试覆盖了部分删除场景，但 `getChildDoms` 内部遍历逻辑未被验证。

## Risks and Open Questions

### Risk 1: `getChildDoms` 调用链复杂
`getChildDoms` 仅在 `commitInstanceDom` 的 `deletions` 循环中被调用。需要构造一个包含标记删除的组件树，可能需要深入理解 `performUnitOfWork` 的 diff 算法。

**缓解**: 先通过简单的手动构造 `deletions` 集合来测试 `getChildDoms`，而非依赖完整的 diff 流程。

### Risk 2: `onUpdateRequire` 触发条件复杂
`onUpdateRequire` 需要状态变化来触发。当前测试使用 mock 的 Signal，可能需要调整 mock 行为。

**缓解**: 使用真实的 Signal（`createState`）并触发更新，验证 `onUpdateRequire` 被调用。

### Risk 3: jsdom SVG namespace 限制
jsdom 不完全支持 SVG namespace。测试 SVG 时需要注意断言方式。

**已验证**: 现有 SVG 测试通过检查元素存在性而非 `namespaceURI` 来避免 jsdom 限制。

### Open Question: 测试隔离
删除路径测试可能需要修改全局 Signal mock，可能影响其他测试。需要确保测试隔离。
