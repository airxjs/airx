## 任务摘要

定义 SSR API 与类型：如 `createSSRApp` / `renderToString` / `hydrate` 等候选接口、上下文类型、注水协议与错误模型；评估与现有 `createApp`、`Router`、`Signal` 的兼容性与破坏性风险。接口命名与类型设计优先直观可懂、易于测试与迁移。

## Design Guidelines and Specs

### 架构约束
- **architecture.md#L52-L68**: 模块划分清晰，`render/server/` 为 SSR 目录，`render/browser/` 为 Browser 渲染目录
- **architecture.md#L70-L85**: 公开 API 以 `source/index.ts` 为准，legacy API 已在 0.7.0 移除
- **architecture.md#L7**: 不提供路由、全局状态方案、打包链路与服务端框架集成层

### API 设计原则
- **ssr-api-spec-template.md**: API 简单直观、可自解释、类型约束明确、错误模型可预测
- **ssr-architecture.md**: 定义了 SSR 总体架构、Signal 序列化机制、hydrate 流程

### 与现有 API 对齐
- **createApp**: 返回 `AirxApp`，有 `.mount()` 方法
- **createSSRApp**: 应返回 `SSRApp`，有 `.renderToString()` 方法
- 保持相似的 API 风格和命名模式

### 类型契约设计（基于 ssr-architecture.md）

```typescript
// SSR 应用入口
export interface SSRApp {
  renderToString(): Promise<SSRResult>
}

// SSR 渲染结果
export interface SSRResult {
  html: string
  stateSnapshot: StateSnapshot | null
  version: string
  errors: Error[]
}

// 客户端激活选项
export interface HydrateOptions {
  stateSnapshot?: StateSnapshot
  forceReset?: boolean
}
```

### Signal 兼容性
- `Signal.State` 需要支持序列化（通过 `get()` 获取值）
- `Signal.Computed` 和 `Signal.Effect` 在 SSR 下行为需要明确
- hydrate 时需要 `Signal.State.getById()` 机制

## Implementation Plan

### API 签名设计

| 接口 | 签名 | 说明 |
|------|------|------|
| `createSSRApp` | `(element: AirxElement, options?: SSRAppOptions) => SSRApp` | SSR 应用工厂 |
| `renderToString` | `(app: SSRApp) => Promise<SSRResult>` | SSRApp 实例方法 |
| `hydrate` | `(element: AirxElement, container: Element, options?: HydrateOptions) => AirxApp` | 客户端激活 |

### 类型定义位置

```
source/render/server/
├── index.ts           # 公开导出
├── ssr-app.ts         # createSSRApp 实现
├── ssr-types.ts       # SSR 类型定义（新增）
├── serialize.ts       # 状态序列化
└── context.ts         # SSR 上下文
```

### 与现有 createApp 的兼容性

| 方面 | createApp | createSSRApp |
|------|-----------|--------------|
| 入参 | AirxElement | AirxElement + SSRAppOptions |
| 返回 | AirxApp | SSRApp |
| 挂载 | .mount(dom) | .renderToString() |
| 插件 | .use(plugin) | 不支持（SSR 无插件扩展） |
| 生命周期 | onMounted/onUnmounted | 不支持（SSR 无挂载概念） |

### 破坏性风险评估

- **低风险**: 新增 API，不修改现有 API
- **无破坏性变更**: 不影响现有 `createApp` 用户

## Risks and Open Questions

### 已识别风险
1. **Signal ID 机制缺失**: 当前 `Signal.State` 无 `id` 字段，需要新增
2. **hydrate 时机**: 事件监听器绑定时机需明确
3. **生命周期行为**: `onMounted` 在 SSR 下是否触发需决策

### 开放问题
1. **hydrate 返回值**: 返回 `AirxApp` 还是 `void`？
2. **错误类型**: SSR 错误是否需要区分错误类型（渲染错误 vs 系统错误）？
3. **stateSnapshot 注入位置**: 放在 `<head>` 还是 `<body>` 内？
