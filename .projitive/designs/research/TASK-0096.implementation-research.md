# TASK-0096 Implementation Research Brief

## Task
建立 Airx 性能回归检测机制

## Summary
当前 benchmark 结果已手动更新多次，建议建立自动化性能回归检测，在 CI 中对关键指标（Signal.State set/get、Signal.Computed derivation、DOM 操作）设置阈值，防止性能退化。

- `benchmark-results.json` - 性能基线数据文件，存储各指标的基准值
- `vitest.benchmark.config.ts` - Vitest benchmark 配置文件
- `.github/workflows/check.yml` - 现有 CI 工作流，仅包含 lint/typecheck/build

## Code Architecture and Implementation Findings

### 现有 Benchmark 基础设施

1. **Benchmark 配置文件** (`vitest.benchmark.config.ts`):
   - 测试环境: `jsdom`
   - 测试文件: `benchmark/**/*.ts`
   - 排除: `benchmark/**/*.test.ts`

2. **Benchmark 数据文件** (`benchmark-results.json`):
   - 包含 7 个指标的基准数据: `Signal.State set/get`, `Signal.Computed derivation`, `Signal.Watcher`, `Simple div element`, `Element with 5 children`, `Deep tree`, `Wide tree`
   - 每个指标包含: `mean`, `stddev`, `median`, `min`, `max`, `iterations`, `opsPerSec`

3. **CI 工作流** (`.github/workflows/check.yml`):
   - 触发: push to main, PR to any branch
   - Jobs: `lint` (npm run typecheck, lint, build)
   - **缺失**: 没有运行 benchmark 或性能检查

### Benchmark 运行方式

```bash
# 运行所有 benchmark
npm run benchmark  # 或: npx vitest run --config vitest.benchmark.config.ts
```

## Implementation Plan

### 1. 创建 Performance Regression CI Workflow

新建 `.github/workflows/benchmark.yml`:

```yaml
name: Performance Benchmark

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  benchmark:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Run benchmarks
        run: npm run benchmark
      
      - name: Compare with baseline
        run: node scripts/compare-benchmark.js
      
      - name: Upload benchmark results
        uses: actions/upload-artifact@v4
        with:
          name: benchmark-results
          path: benchmark-results.json
```

### 2. 创建基准比较脚本

新建 `scripts/compare-benchmark.js`:
- 读取当前 `benchmark-results.json`
- 与基线比较（可存储在 repo 根目录或作为 artifact 上传）
- 对每个指标检查性能退化阈值（默认 10%）
- 如果退化超过阈值，CI 失败并报告详情

### 3. 性能阈值配置

在 `package.json` 中添加 benchmark 配置:

```json
{
  "benchmark": {
    "threshold": 0.1,
    "baseline": "benchmark-results.json"
  }
}
```

### 4. 更新 benchmark-results.json 为可追踪的基线

将当前结果 commit 为初始基线，后续更改会触发比较。

## Risks and Open Questions

### 已知风险

1. **CI 运行时间增加**: benchmark 测试需要额外时间，可能影响 CI 总耗时
2. **环境差异**: 不同 runner 硬件差异可能导致结果波动
3. **阈值设置**: 10% 阈值是否合理需要根据实际数据调整

### 解决方案

1. **阈值调整**: 首次启用建议用 20% 阈值，稳定后逐步降低
2. **环境隔离**: 使用固定版本 Node.js 和依赖版本
3. **基线更新**: main 分支成功时自动更新基线，避免累积退化

### 待确认

1. 是否需要 `benchmark-results.json` 作为初始基线 commit 到 repo?
2. 基线更新策略: 手动更新 vs 自动更新?
3. 是否需要 Slack/邮件通知性能退化?
