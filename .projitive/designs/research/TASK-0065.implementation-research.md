# TASK-0065 Implementation Research Brief

- Task: hydrate() stub 运行时警告与 API 稳定性标注
- Summary: 当前 hydrate() 是 no-op stub，用户调用后不会有任何效果也不报错，这会误导下游。需：1）在 hydrate() 被调用时输出明确的运行时警告（console.warn），说明当前为 stub 状态；2）参照 api-stability-audit.md，为所有实验性 API（hydrate、plugin、renderToHTML）添加 @alpha 或 @experimental 标签；3）确保警告信息指向 0.8.x roadmap，让用户知道这是计划中的功能。验收标准：调用 hydrate() 时控制台有明确警告，ts 类型注解中有 @experimental 标记。

## Code Architecture and Implementation Findings

### 实验性 API 识别

以下 API 被认定为实验性（`@experimental`）:

| API | 文件 | 类型 | 位置 |
|-----|------|------|------|
| `SSRApp.hydrate()` | server.ts | Method | server.ts#L348 |
| `hydrate()` (top-level) | server.ts | Function | server.ts#L373 |
| `HydrateOptions` | browser/hydrate.ts | Interface | hydrate.ts#L9 |
| `StateSnapshot` | browser/hydrate.ts | Interface | hydrate.ts#L25 |
| `HydratedApp` | browser/hydrate.ts | Interface | hydrate.ts#L37 |

### 关键发现

1. **`plugin()` 和 `renderToHTML()` 不是实验性** — api-stability-audit.md Issue 1 已解决，这两个方法已从 @deprecated WIP 状态解除，标记为正式稳定 API。不应添加 @experimental 标签。

2. **当前 hydrate 实现不是纯 stub** — `SSRApp.hydrate()` 实际上调用了 `clientHydrate()`，已有部分实现。但作为实验性 API，仍需警告。

3. **console.warn vs logger.warn** — 选择 console.warn 而非内部 logger，因为 hydrate 警告需要触达最终用户，而 logger 默认级别可能不输出。
