### Key Findings

1. **No minification** — `tsc` only emits raw ES modules. The ~17KB UMD bundle is unminified
2. **Source maps inflate output** — .js.map files add ~30% to total size
3. **Reconciliation logic is the largest module** — `common.js` (21KB) contains the virtual DOM diffing algorithm
4. **Server renderer is substantial** — `server.js` (12KB) for SSR-only usage adds significant weight
5. **Hydration is separate** — `hydrate.js` (7.5KB) could potentially be split for SSR users who don't need hydration
6. **ES module re-exports add overhead** — `output/index.js` re-exports from subpaths, preventing direct usage

## Implementation Plan

### Phase 1: Bundle Audit (this task)

**Deliverable: Bundle analysis report**

Current measurements:
- **ESM entry** (`output/index.js` + submodules): ~50KB with source maps, ~29KB estimated minified
- **UMD bundle** (`output/index.umd.cjs`): ~17KB unminified, ~10KB estimated minified
- **Largest module**: `render/basic/common.js` (21KB / 38%)

### Phase 2: Optimization Recommendations

Based on analysis, prioritized optimization opportunities:

| Priority | Optimization | Potential Savings | Effort |
|----------|--------------|-------------------|--------|
| P1 | Add minification step (esbuild/terser) | ~40% size reduction | Low |
| P1 | Remove source maps from npm package | ~30% size reduction | Low |
| P2 | Tree-shaking audit (ensure dead code elimination) | ~5-10% | Medium |
| P2 | Split SSR and browser renders | Allow conditional loading | High |
| P3 | Investigate hydrate.js separation | SSR users skip 7.5KB | Medium |

### Specific Optimization Targets

**Before (current)**:
- ESM total: ~50KB (with source maps)
- UMD: ~17KB (unminified)
- Minified estimate: ~29KB ESM, ~10KB UMD

**After (target for 0.9.0)**:
- ESM total: <30KB (with source maps) — 40% reduction
- UMD minified: <6KB — 40% reduction
- No source maps in published package

## Risks and Open Questions

1. **Risk**: Removing source maps may hurt downstream debugging — mitigate by keeping them in a separate `-debug` package or GitHub releases
2. **Risk**: Minification may change behavior of some edge cases — mitigate by running full test suite post-minification
3. **Open Question**: Should hydration code be split into a separate entry point? SSR-only users pay 7.5KB for hydrate they don't need
4. **Open Question**: Should we consider a "light" build that excludes SSR for browser-only usage?
5. **Assumption**: The 0.9.0 roadmap prioritizes DX improvements; bundle optimization aligns with this goal
