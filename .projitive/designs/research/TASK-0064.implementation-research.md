## Code Architecture and Implementation Findings

### 当前 Hydrate 实现状态

**browser/hydrate.ts 已有骨架**:
- `hydrate(element, container, options)` — 完整的函数签名
- `connectInstanceTreeToDom()` — 连接实例树到现有 DOM，已调用 `updateDomProperties`
- `setupReactiveUpdates()` — 设置响应式更新调度
- `HydrateOptions`, `StateSnapshot`, `HydratedApp` 接口已定义

**关键缺失**:
1. `renderToString` 不序列化 Signal 状态，输出 HTML 无 stateSnapshot
2. `hydrate()` 不从 HTML 读取 stateSnapshot（`options?.stateSnapshot` 未被使用）
3. `connectInstanceTreeToDom` 事件绑定逻辑正确（通过 BasicLogic.updateDom），但缺少 mismatch 检测

### 当前 renderToString 流程

```
createSSRApp(element)
  └── renderToString()
        └── render(appContext, rootElement, onComplete)
              ├── 构建 Instance 树（performUnitOfWork）
              ├── commitDom() 生成 HTML 字符串
              └── onComplete(html) — 直接返回 HTML，无 stateSnapshot
```

### 当前 hydrate 流程

```
hydrate(element, container, options)
  ├── 构建 Instance 树（performUnitOfWork）
  ├── connectInstanceTreeToDom() — 连接 DOM，调用 updateDomProperties
  │     └── BasicLogic.updateDom() — 绑定事件（prevProps={} 时正确添加）
  └── setupReactiveUpdates() — 启动响应式调度
```

### 问题分析

**问题 1: Signal 状态序列化缺失**
- `render()` 调用 `onComplete(context.rootInstance.domRef?.toString() || '')` 直接返回 HTML
- 无 Signal 状态收集机制
- signal-polyfill 的 Signal.State 无内置全局追踪

**问题 2: hydrate 不使用 stateSnapshot**
- `hydrate()` 函数签名有 `options?: HydrateOptions`
- 但函数体中完全没有读取 `options?.stateSnapshot`
- 没有 `restoreSignalStates()` 函数

**问题 3: Mismatch 检测缺失**
- `connectInstanceTreeToDom` 使用简单的标签名匹配
- 匹配失败时只是返回 null，不警告用户

## Implementation Plan

### Phase 1: Signal 状态序列化（SSR）

**修改 `source/render/server/server.ts`**:

1. 在 `render()` 函数外创建 Signal 追踪机制：
   ```typescript
   // SSR 期间收集 Signal 状态的注册表
   const ssrSignalRegistry = new Map<string, unknown>()
   
   export function registerSSRSignal(id: string, value: unknown) {
     ssrSignalRegistry.set(id, value)
   }
   
   export function getSSRStateSnapshot(): StateSnapshot {
     return {
       signals: Object.fromEntries(ssrSignalRegistry),
       version: packageVersion,
       timestamp: Date.now()
     }
   }
   ```

2. 修改 `render()` 的 `onComplete` 回调：
   ```typescript
   onComplete(context.rootInstance.domRef?.toString() || '')
   // 改为：
   const html = context.rootInstance.domRef?.toString() || ''
   const stateSnapshot = getSSRStateSnapshot()
   const stateScript = `<script type="airx/ssr-state">${JSON.stringify(stateSnapshot)}</script>`
   onComplete(html + stateScript)
   ```

### Phase 2: hydrate 状态恢复

**修改 `source/render/browser/hydrate.ts`**:

1. 添加 `readStateSnapshot()` 函数：
   ```typescript
   function readStateSnapshot(container: HTMLElement): StateSnapshot | null
   ```

2. 添加 `restoreSignalStates()` 函数：
   ```typescript
   function restoreSignalStates(snapshot: StateSnapshot): void
   ```

3. 在 `hydrate()` 中调用状态恢复：
   ```typescript
   if (!options?.forceReset && options?.stateSnapshot) {
     restoreSignalStates(options.stateSnapshot)
   }
   ```

### Phase 3: Mismatch 检测

**修改 `source/render/browser/hydrate.ts`**:

1. 在 `connectInstanceTreeToDom` 中添加 mismatch 检测：
   ```typescript
   function detectMismatch(instance: Instance, dom: HTMLElement): HydrateMismatch | null
   ```

2. 当检测到 mismatch 时输出警告

### Phase 4: Counter 演示验证

需要验证的端到端流程：
1. SSR: `createSSRApp(Counter).renderToString()` 输出包含 stateSnapshot script
2. Hydrate: `hydrate(element, container)` 能恢复 Counter 状态
3. 交互: 点击按钮，Counter 正确递增

## Risks and Open Questions

1. **Signal ID 机制**: 当前实现中 Signal 没有稳定 ID，序列化时使用运行时生成的 key
2. **Hydration 边界**: 哪些组件需要在 hydrate 时激活，哪些可以延迟（首版全部激活）
3. **Mismatch 恢复策略**: DOM 与组件树不一致时的行为（当前：warn + 继续）
