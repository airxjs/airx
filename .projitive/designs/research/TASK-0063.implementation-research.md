## Code Architecture and Implementation Findings

### 当前 Hydrate 实现状态

**server.ts 中的 hydrate**:
- `SSRApp.hydrate()` — 调用 `clientHydrate(rootElement, container, options)`
- 顶级 `hydrate()` — 调用 `app.hydrate(container, options)`

**browser/hydrate.ts 中的实现**:
- `hydrate(element, container, options)` 函数已有完整骨架

## Implementation Plan

### 需要新增的模块

1. **Signal 序列化器** — 在 server.ts 的 render 流程中收集 Signal 状态，生成 StateSnapshot
2. **StateSnapshot 注入** — 将 StateSnapshot 注入到 HTML 中（`<script type="airx-state">`）
3. **状态恢复** — 在 hydrate() 中从 DOM 读取 StateSnapshot 并恢复 Signal 树
4. **事件委托增强** — 完善 connectInstanceTreeToDom 的事件绑定

### 验证计划

- SSR renderToString → 检查输出 HTML 包含 stateSnapshot
- hydrate → 检查 Signal 状态正确恢复，事件正确绑定
- 端到端 Counter 演示 → 服务端渲染 + 客户端激活全程工作

## Risks and Open Questions

1. **Signal ID 机制**: Signal 需要有稳定 ID 才能序列化/恢复，当前 Signal 实现可能缺少
2. **Hydration 边界**: 哪些组件需要在 hydrate 时激活，哪些可以延迟
3. **Mismatch 检测**: DOM 与组件树不一致时如何检测和处理
