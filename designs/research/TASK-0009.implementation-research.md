## 2. Code Architecture and Implementation Findings

### 2.1 signal.ts 未覆盖行分析

**未覆盖行**: `source/signal/signal.ts#L19-25`

```typescript
function getSignal() {
  const globalSignal: typeof Polyfill = globalNS['Signal']
  if (globalSignal == null) throw new Error('Signal is undefined')
  if (firstSignal == null) firstSignal = globalSignal      // ← 行 19
  if (firstSignal !== globalSignal) throw new Error('Signal have multiple instances')  // ← 行 20
  return globalSignal
}
```

**行 19 分析**: 仅在 `firstSignal == null` 时执行（模块首次加载时）。现有测试在 beforeEach 中设置了 `globalThis.Signal`，但 `firstSignal` 作为模块级变量在首次 `getSignal()` 调用时被设置，之后不再进入此分支。

**行 20 分析**: 检测多实例冲突。当两次加载了不同版本的 signal-polyfill 时触发。

**关键约束**:
- `firstSignal` 是模块级单例，Vitest 默认 `isolateModules` 无法重置它
- `globalNS` 在模块评估时捕获，不随 `globalThis.Signal` 赋值变化
- signal-polyfill 是 npm 包，无法直接在测试中制造"不同实例"

### 2.2 覆盖策略分析

**方案 A: vi.isolateModules + mock signal-polyfill（推荐）**
```typescript
it('should throw when multiple Signal instances detected', async () => {
  // 1. 用 vi.mock 替换 signal-polyfill（返回 mock Signal A）
  // 2. vi.isolateModules() 内重新 import signal 模块 → firstSignal = A
  // 3. 在外层替换 globalThis.Signal = mock Signal B
  // 4. 再次 isolateModules → getSignal() 读到 B，B !== A → 抛出
})
```

**方案 B: 直接访问模块内部状态**
- 通过 `vi.mock` 拦截 `signal-polyfill` 返回不同 mock class
- 需要 `vi.resetModules()` 配合

**方案 C: 接受不可达（保守）**
- 行 19-20 是防御性代码，真实场景几乎不会触发
- 覆盖率从 80.55% → 82-85% 已足够
- 专注于 browser.ts/server.ts 的集成测试更有价值
