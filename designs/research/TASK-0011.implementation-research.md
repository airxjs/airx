## Code Architecture and Implementation Findings

### 1. Public API 到运行时入口

- [source/app/app.ts#L4-L44](../../source/app/app.ts#L4-L44): `createApp` 创建 `PluginContext`，把函数组件入口标准化为 `AirxElement`，并分发到 `browserRender` / `serverRender`。
- [source/render/index.ts](../../source/render/index.ts): render 层只暴露 hooks、browser/server renderer 与插件上下文。

结论：App 层足够薄，当前主要职责是装配，不应继续承载更多状态或路由语义。

### 2. Element 与组件模型

- [source/element/element.ts](../../source/element/element.ts): `createElement` 负责 children 归一化；`component` 只是类型辅助；`createErrorRender` 是运行时兜底。

结论：Element 层模型稳定，但 README 与架构文档此前仍保留 legacy/过时描述，需要同步修正。

### 3. 协调与执行主路径

- [source/render/basic/common.ts#L191-L392](../../source/render/basic/common.ts#L191-L392): `reconcileChildren` 基于 key 或内部 index key 协调 children，并通过 plugin 决定是否复用 instance。
- [source/render/basic/common.ts#L402-L533](../../source/render/basic/common.ts#L402-L533): `performUnitOfWork` 负责组件执行、Signal 依赖收集、children 标准化与深度优先遍历推进。

结论：Airx 已经具备 Fiber-like 的增量执行骨架，原架构文档中“待实现”的描述已过时。

### 4. 浏览器调度与 DOM 提交

- [source/render/browser/browser.ts#L14-L240](../../source/render/browser/browser.ts#L14-L240): 浏览器渲染维护 `nextUnitOfWork`、`needCommitDom` 与提交流程。
- [source/render/browser/browser.ts#L217-L239](../../source/render/browser/browser.ts#L217-L239): 调度循环原先无条件递归请求 `requestIdleCallback`，即使没有剩余工作也持续空转。

结论：浏览器调度器是本轮最值得立即修复的实现问题，因为它影响 CPU 空转、兼容性与后续性能分析准确性。

### 5. 插件与注入系统

- [source/render/basic/plugins/internal/basic/basic.ts#L55-L173](../../source/render/basic/plugins/internal/basic/basic.ts#L55-L173): `BasicLogic` 负责 props 对比、DOM 更新和实例复用判断。
- [source/render/basic/plugins/internal/inject/inject.ts#L4-L27](../../source/render/basic/plugins/internal/inject/inject.ts#L4-L27): `InjectSystem` 通过 `injectedMap` 检测依赖注入值变化。

结论：插件是当前最明确的扩展点，但 DOM 更新策略与 browser/server 提交流程仍然分散，后续应收敛为单一实现源。

## Priority Findings

1. Browser 调度存在空转问题，需要只在存在剩余工作时继续排队，并提供 `requestIdleCallback` 降级路径。
2. Browser / Server 的 `commitDom` 实现高度重复，后续修改易分叉。
3. 架构文档落后于 0.7.0 实现，尤其是 legacy API、Fiber-like 执行路径和版本基线。
4. render 相关测试对 `performUnitOfWork` 过度 mock，尚未覆盖真实 Signal 集成与注入变更路径。
5. Signal watcher 的微任务收集逻辑仍需要后续专项验证。
